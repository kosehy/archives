# S3 Files와 S3의 변화하는 모습

원문: https://www.allthingsdistributed.com/2026/04/s3-files-and-the-changing-face-of-s3.html
아카이브: 2026-04-09
언어: Korean
유형: 한국어 번역본

## 핵심 내용

이 글은 S3가 단순한 객체 스토리지에서, 테이블과 벡터를 거쳐 이제 파일 인터페이스까지 품으려는 방향으로 어떻게 진화하고 있는지를 설명한다. 중심 주제는 데이터 자체보다 데이터에 접근하는 방식이 도구와 워크로드마다 달라질 때 생기는 마찰, 즉 "data friction"이다.

글은 해바라기 유전체 분석 사례로 시작한다. 생물학자들은 대규모 병렬 계산을 위해 클라우드를 활용하고 싶었지만, 실제 도구들은 로컬 리눅스 파일시스템을 전제로 만들어져 있었다. S3는 병렬성, 비용, 내구성 면에서 훌륭했지만, 파일시스템 기반 도구와 직접 맞물리지 않아 데이터를 계속 복사하고 동기화해야 했다. 저자는 이런 문제가 생명과학뿐 아니라 미디어, 머신러닝 프리트레이닝, 반도체 설계, 과학 계산 등 여러 분야에서 반복된다고 말한다.

이 문제는 에이전트 시대에 더 커진다고 본다. 에이전트 기반 개발은 소프트웨어 작성 비용을 낮추고, 더 많은 도메인 전문가가 직접 애플리케이션을 만들 수 있게 해주지만, 동시에 데이터에 손쉽게 붙을 수 있는 스토리지 계층의 중요성을 더 키운다. 애플리케이션은 점점 더 빨리 생겼다가 사라지지만 데이터는 오래 남기 때문에, 저장소는 데이터를 안전하게 보관하는 것뿐 아니라 다양한 도구와 애플리케이션이 쉽게 붙고 떨어질 수 있도록 해주는 추상화 계층이어야 한다는 주장이다.

이 문제를 풀기 위한 S3의 최근 진화로 글은 먼저 S3 Tables와 S3 Vectors를 설명한다.

- S3 Tables는 Iceberg 기반의 테이블을 S3의 일급 데이터 프리미티브로 다루려는 시도다.
- S3 Vectors는 유사도 검색용 벡터 인덱스를 S3 특유의 탄력성과 비용 구조로 제공하려는 시도다.
- 두 기능 모두 핵심은 스토리지 자체를 더 높은 수준의 데이터 추상화로 제공해, 고객이 데이터 작업보다 도구와 운영 마찰에 덜 묶이게 하려는 데 있다.

그리고 이번 글의 중심인 S3 Files가 등장한다. S3 Files는 Amazon EFS를 S3와 통합해, 기존 S3 데이터를 네트워크 파일시스템처럼 직접 접근할 수 있게 해주는 기능으로 소개된다. 즉 EC2, 컨테이너, Lambda에서 S3 버킷이나 prefix를 파일시스템처럼 마운트하고, 파일 인터페이스로 읽고 수정한 결과를 다시 S3 객체 세계와 연결하는 방식이다.

하지만 저자는 파일과 객체를 하나의 저장 시스템 안에서 자연스럽게 결합하는 일이 매우 어렵다고 강조한다. 파일은 운영체제 수준의 풍부한 공유·수정 의미론을 가진 반면, 객체는 불변성과 전체 객체 생성/교체 이벤트 중심의 모델을 전제로 한다. 파일 쪽은 중간 덮어쓰기, mmap, 세밀한 동시 접근이 자연스럽고, 객체 쪽은 버전 관리, 해시 검증, whole-object notification에 맞춰 설계돼 있기 때문이다. 그래서 초기 설계는 "둘의 장점만 합치자"는 단순한 생각에서 출발했지만, 실제로는 양쪽 모두에게 불편한 타협만 낳았다고 회고한다.

글의 후반부는 바로 이 난제를 어떻게 재정의했는지, 그리고 파일/객체 융합을 고객에게 단순하게 보이게 만들기 위해 얼마나 복잡한 설계 고민이 필요했는지를 다룬다. 즉 핵심 메시지는 단순히 S3 Files 출시 소식이 아니라, 데이터 API의 차이 자체가 현대 소프트웨어 생산성과 실험 속도를 가로막는 병목이 되고 있으며, S3가 이를 줄이는 방향으로 점점 더 많은 데이터 타입을 직접 품으려 한다는 데 있다.

## 한 줄 요약

이 글은 S3 Files를 통해 S3가 객체 스토리지를 넘어 파일 접근까지 흡수하면서, 도구별 데이터 접근 마찰을 줄이는 더 넓은 데이터 플랫폼으로 진화하고 있다고 설명한다.
