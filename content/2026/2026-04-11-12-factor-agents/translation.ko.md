# 12-factor agents

원문: https://github.com/humanlayer/12-factor-agents
저장일: 2026-04-11
번역 메모: 저장 가능한 GitHub 리포지토리 소개 페이지를 기준으로 자연스러운 한국어 흐름에 맞춰 정리한 번역본.

이 프로젝트는 [12 Factor Apps](https://12factor.net/)의 문제의식을 AI 에이전트 소프트웨어로 확장하려는 시도다. 소스는 공개되어 있고, 작성자는 피드백과 기여를 환영한다고 밝힌다.

저자 Dex는 오랫동안 AI 에이전트를 다뤄왔고, plug-and-play 계열부터 minimalist 계열, production-grade를 표방하는 여러 프레임워크까지 폭넓게 써봤다고 말한다. 또한 YC 안팎의 강한 창업자들과 대화하며, 실제 고객 대상 프로덕션 에이전트에서는 프레임워크보다 직접 스택을 짜는 경우를 더 많이 봤다고 한다.

그가 놀랐던 점은, 시장에서 “AI Agent”라고 불리는 많은 제품이 실제로는 그렇게까지 agentic하지 않다는 것이다. 상당수는 결정론적 코드가 중심이고, LLM 단계가 적절한 지점에만 들어가 사용자 경험을 마법처럼 보이게 만든다.

이 관점에서 저자는 좋은 에이전트가 단순히 “프롬프트 하나, 도구 몇 개, 목표 달성까지 루프” 같은 패턴으로 설명되지 않는다고 본다. 오히려 대부분은 결국 소프트웨어 그 자체에 가깝다. 그래서 그는 다음 질문을 던지며 `12-factor agents`를 정리하기 시작했다.

LLM이 앞으로도 계속 강력해지더라도, LLM 기반 소프트웨어를 더 신뢰할 수 있고, 더 확장 가능하며, 더 유지보수하기 쉽게 만드는 핵심 엔지니어링 원칙은 남아 있을 것이다.

## 구성

- 소프트웨어의 간단한 역사
- Factor 1: 자연어를 도구 호출로 바꾸기
- Factor 2: 프롬프트를 스스로 소유하기
- Factor 3: 컨텍스트 윈도우를 스스로 소유하기
- Factor 4: 도구는 결국 구조화된 출력이다
- Factor 5: 실행 상태와 비즈니스 상태를 통합하기
- Factor 6: 단순한 API로 시작/일시정지/재개하기
- Factor 7: 도구 호출로 인간과 접촉하기
- Factor 8: 제어 흐름을 스스로 소유하기
- Factor 9: 오류를 컨텍스트 윈도우 안으로 압축하기
- Factor 10: 작고 집중된 에이전트
- Factor 11: 어디서든 트리거되고, 사용자가 있는 곳에서 만나라
- Factor 12: 에이전트를 stateless reducer로 만들기

## 핵심 문제의식

이 가이드는 소프트웨어를 directed graph로 바라보는 시각에서 출발한다. 한때 에이전트는 엔지니어가 모든 단계와 예외 처리를 직접 코딩하지 않고, LLM에게 목표와 전이 가능성만 주면 경로를 실시간으로 찾아낼 수 있을 것이라는 기대를 불러왔다.

리포지토리는 전형적인 에이전트 루프를 다음처럼 설명한다.

```ts
initial_event = {"message": "..."}
context = [initial_event]
while True:
  next_step = await llm.determine_next_step(context)
  context.append(next_step)

  if (next_step.intent === "done"):
    return next_step.final_answer

  result = await execute_step(next_step)
  context.append(result)
```

하지만 저자는 이 접근이 기대만큼 잘 작동하지 않는다고 본다. 많은 팀이 프레임워크를 써서 빠르게 70~80% 수준의 품질까지는 가지만, 실제 고객 대상 기능에 필요한 수준까지 끌어올리려면 결국 프롬프트, 컨텍스트, 흐름, 실행 구조를 깊이 이해하고 통제해야 한다는 것이다. 그 과정에서 적지 않은 팀이 처음부터 다시 짜게 된다.

## 중심 주장

저자가 여러 라이브러리를 뒤지고 많은 창업자들과 일하며 얻은 결론은 이렇다.

- 좋은 에이전트를 만드는 핵심 원칙은 분명히 존재한다.
- 프레임워크에 전면적으로 올라타며 사실상 그린필드 재작성으로 가는 길은 오히려 비효율적일 수 있다.
- 고품질 AI 소프트웨어를 고객 손에 가장 빨리 쥐여주는 방법은, 에이전트 구축에서 나온 모듈형 개념을 기존 제품에 흡수하는 것이다.
- 숙련된 소프트웨어 엔지니어라면, 꼭 깊은 AI 배경이 없어도 이런 개념을 적용할 수 있다.
