# 장시간 애플리케이션 개발을 위한 하니스 설계

원문: https://www.anthropic.com/engineering/harness-design-long-running-apps
저장일: 2026-04-11
번역 메모: 직역보다 자연스러운 한국어 흐름을 우선해 다듬은 번역본.

이 글은 Anthropic Labs의 Prithvi Rajasekaran이, Claude로 더 좋은 프론트엔드 디자인을 만들고 인간 개입 없이 완전한 애플리케이션을 구축하게 만드는 실험 과정을 정리한 글이다. 출발점은 이전의 frontend design skill과 장기 실행 코딩 에이전트 하니스였고, 그 작업을 통해 프롬프트 엔지니어링과 하니스 설계가 분명한 성능 향상을 만들 수 있다는 점은 확인했지만, 결국 일정한 한계도 마주했다.

그 한계를 넘기 위해 저자는 두 가지 상이한 영역, 즉 취향이 중요한 디자인과 검증 가능한 정확성이 중요한 소프트웨어 개발 모두에 적용 가능한 새로운 AI 엔지니어링 접근을 찾았다. GAN(Generative Adversarial Networks)에서 영감을 받아 generator와 evaluator를 분리한 멀티에이전트 구조를 설계했고, “좋은 디자인인가?” 같은 주관적 판단도 평가 가능하도록 세부 기준으로 바꾸는 작업부터 시작했다.

이후 이 방법을 장시간 자율 코딩에 적용하면서, 이전 하니스 작업에서 얻은 두 가지 교훈을 이어받았다. 하나는 빌드를 다룰 수 있는 단위로 분해하는 것, 다른 하나는 세션 간 컨텍스트를 넘길 수 있도록 구조화된 산출물을 사용하는 것이다. 그 결과 planner, generator, evaluator로 이루어진 3에이전트 구조가 만들어졌고, 이 구조는 몇 시간에 걸친 자율 코딩 세션 동안 풍부한 풀스택 애플리케이션을 생성해냈다.

## 왜 순진한 구현은 부족한가

저자는 이전 연구에서 장기 실행 코딩 에이전트의 효과가 하니스 설계에 크게 좌우된다는 점을 이미 보여준 바 있다고 말한다. 그때는 initializer agent가 제품 명세를 작업 목록으로 분해하고, coding agent가 기능을 하나씩 구현한 뒤, 산출물을 넘겨 다음 세션에서 맥락을 이어가게 하는 구조를 썼다. 개발자 커뮤니티도 비슷한 방향으로 수렴하고 있으며, 예를 들어 hooks나 scripts로 에이전트를 계속 반복 루프에 두는 “Ralph Wiggum” 방식 같은 접근도 언급된다.

그럼에도 복잡한 과제에서는 에이전트가 시간이 갈수록 궤도를 이탈하는 문제가 남아 있었다. 저자는 여기서 두 가지 흔한 실패 모드를 지적한다.

첫째는 컨텍스트가 길어질수록 모델이 일관성을 잃는 문제다. 어떤 모델은 자신의 컨텍스트 한계에 가까워졌다고 느끼면 작업을 너무 일찍 마무리하려는 “context anxiety”도 보인다. 이 문제는 컨텍스트를 완전히 비우고 새 에이전트를 시작하는 context reset과, 이전 상태와 다음 단계가 담긴 구조화된 handoff를 결합하면 완화할 수 있다.

이 방식은 같은 에이전트가 요약된 기록을 들고 계속 가는 compaction과 다르다. compaction은 연속성을 보존하지만 깨끗한 초기 상태를 주지는 못하기 때문에 context anxiety가 계속 남을 수 있다. 반대로 reset은 깨끗한 출발점을 주지만, 그만큼 handoff 산출물이 충분한 상태 정보를 담고 있어야 한다. 실제 테스트에서 Claude Sonnet 4.5는 context anxiety가 강해서 compaction만으로는 충분하지 않았고, 그래서 context reset이 하니스 핵심 요소가 되었다. 다만 그 대가로 오케스트레이션 복잡도, 토큰 오버헤드, 지연 시간이 늘어난다.

둘째 문제는 self-evaluation이다. 에이전트는 자신이 만든 결과물을 평가하라고 하면, 사람 눈에는 평범하거나 부족해 보여도 자신 있게 칭찬하는 경향이 있다. 디자인처럼 정답 검증이 어려운 주관적 작업에서는 이 문제가 특히 심하다. 하지만 검증 가능한 결과가 있는 작업에서도, 자신의 산출물에 대해 지나치게 후한 판단을 내리는 문제가 성능을 떨어뜨린다. 그래서 작업하는 에이전트와 평가하는 에이전트를 분리하는 것이 강한 레버가 된다. evaluator 역시 결국 LLM이지만, generator가 자기 작업을 비판적으로 보게 만드는 것보다, 독립 evaluator를 회의적으로 튜닝하는 편이 훨씬 쉽다.

## 프론트엔드 디자인: 주관적 품질을 채점 가능하게 만들기

저자는 self-evaluation 문제가 가장 잘 드러나는 프론트엔드 디자인부터 실험을 시작했다. 별다른 개입이 없으면 Claude는 기술적으로는 문제 없지만 시각적으로 평범한 레이아웃으로 수렴하는 경향이 있었다.

여기서 두 가지 통찰이 프론트엔드 하니스 설계를 이끌었다.

1. 미적 감각을 완전히 점수화할 수는 없지만, 디자인 원칙과 선호를 담은 평가 기준을 통해 더 나은 방향으로 유도할 수 있다.
2. 프론트엔드 생성과 평가를 분리하면, 더 강한 결과물을 향해 반복적으로 밀어주는 피드백 루프를 만들 수 있다.

저자는 generator와 evaluator 모두에게 네 가지 기준을 넣었다.

- 디자인 품질
- 독창성
- 완성도(craft)
- 기능성

이 가운데 디자인 품질과 독창성에 더 높은 비중을 두었다. Claude는 기본적으로 craft와 기능성은 꽤 잘 충족했지만, 디자인과 독창성은 자주 밋밋했기 때문이다. 기준은 전형적인 “AI 슬롭” 패턴을 명시적으로 감점하도록 설계됐다.

evaluator는 상세한 few-shot 예시로 보정되었고, Claude Agent SDK와 Playwright MCP를 사용해 실제 페이지를 탐색하고 평가했다. 이 피드백은 다시 generator에게 돌아갔고, 5회에서 15회까지 반복하면서 때로는 4시간 가까이 걸리는 루프가 만들어졌다. 어떤 경우에는 점진적 개선이 아니라, 완전히 다른 미학적 방향으로 점프하는 변화도 나타났다.

## 풀스택 코딩으로 확장하기

이후 저자는 generator-evaluator 루프를 풀스택 애플리케이션 개발에도 적용했다.

### 아키텍처

기존 장기 실행 하니스는 initializer, coding agent, 그리고 세션 사이의 context reset으로 구성되어 있었다. 새로운 하니스에서는 Opus 4.5를 바탕으로 planner, generator, evaluator의 3에이전트 구조로 발전했다.

- Planner: 1~4문장 정도의 짧은 프롬프트를 받아 더 풍부한 제품 명세로 확장한다.
- Generator: React, Vite, FastAPI, SQLite(후에는 PostgreSQL) 기반으로 기능을 sprint 단위로 구현하고, sprint 말에 스스로 점검한 뒤 QA에 넘긴다.
- Evaluator: Playwright MCP로 실행 중인 앱을 실제 사용자처럼 눌러보며 UI, API, DB 상태를 점검하고, 제품 깊이, 기능성, 시각 디자인, 코드 품질 기준으로 평가한다.

각 sprint 시작 전에는 generator와 evaluator가 “무엇이 done인지”를 정의하는 sprint contract를 협의했다. 서로의 커뮤니케이션은 파일을 통해 이뤄졌고, generator는 합의된 contract를 기준으로 구현을 진행했다.

### 하니스 실행 결과

첫 버전에서는 Claude Opus 4.5를 사용해, 풀 하니스와 단일 에이전트 시스템을 비교했다. 테스트 과제 중 하나는 레트로 게임 메이커를 만드는 것이었다.

비용과 시간 차이는 매우 컸다.

- 단독 실행: 20분, 9달러
- 전체 하니스: 6시간, 200달러

하지만 결과물 품질 차이도 컸다. 하니스 쪽은 planner가 한 줄짜리 프롬프트를 16개 기능, 10개 sprint에 걸친 상세 명세로 확장했고, 더 나은 시각 언어와 풍부한 에디터, Claude 기반 인앱 보조 기능, 실제로 작동하는 플레이 모드를 만들어냈다. evaluator는 rectangle fill, entity 삭제, animation frame reorder API 라우팅 문제 같은 구체적 버그도 찾아냈다.

저자는 evaluator를 이 수준까지 끌어올리는 데도 많은 튜닝이 필요했다고 말한다. 기본 상태의 Claude는 QA 에이전트로는 부족했고, 실제 문제를 발견해놓고도 “큰 문제는 아니다”라며 통과시키는 경향이 있었다.

## 하니스 단순화 실험

첫 하니스는 고무적이었지만, 동시에 크고 느리고 비쌌다. 그래서 다음 단계는 성능을 잃지 않으면서 단순화할 수 있는지 보는 것이었다. 저자는 모든 하니스 구성 요소는 “모델이 혼자서는 못 하는 것”에 대한 가정을 담고 있으며, 이 가정은 자주 점검해야 한다고 본다. 틀릴 수도 있고, 모델 발전으로 금세 구식이 될 수도 있기 때문이다.

그 후에는 구성 요소를 하나씩 제거하며 영향을 검토하는 방식으로 더 체계적인 단순화를 시도했다. 마침 Opus 4.6이 나오면서, 더 나은 계획 능력, 장시간 작업 지속력, 대규모 코드베이스에서의 안정성, 코드 리뷰와 디버깅 능력 향상 덕분에 하니스 구조를 줄일 여지도 생겼다.

### sprint 구조 제거

대표적인 단순화는 sprint 구조를 완전히 제거한 것이다. Opus 4.6에서는 모델이 꼭 sprint 단위 분해 없이도 비교적 일관되게 빌드를 이어갈 수 있었다.

다만 planner와 evaluator는 그대로 유지했다. planner가 없으면 generator는 범위를 너무 좁게 잡았고, evaluator가 없으면 마지막 단계의 결함이 여전히 남았기 때문이다. 다만 모델이 혼자서도 충분히 잘하는 과제에서는 evaluator가 부담만 되는 경우도 있었다.

여기서 저자는 중요한 실무적 결론을 뽑는다. evaluator는 항상 필요한 고정 요소가 아니라, 현재 모델이 혼자서는 안정적으로 처리하지 못하는 과제 구간에서만 비용 대비 가치가 높다.

업데이트된 하니스는 Web Audio API 기반 브라우저형 DAW를 만드는 과제로 테스트됐다. 여전히 약 3시간 50분, 124.70달러가 들었지만, QA는 오디오 녹음이 스텁 상태라는 점, 클립 리사이즈/분할 미구현, EQ 시각화 부재 같은 구체적 공백을 계속 찾아냈다.

## 앞으로의 시사점

저자는 모델이 좋아질수록 어떤 보조 구조는 덜 중요해질 수 있지만, 그렇다고 흥미로운 하니스 조합의 공간이 줄어드는 것은 아니라고 본다. 오히려 그 공간은 계속 이동한다. 따라서 AI 엔지니어는 현실 문제에서 모델의 trace를 직접 읽고, 복잡한 작업에서는 전문화된 에이전트 조합을 실험하며, 새 모델이 나올 때마다 더 이상 필요 없는 구조는 걷어내고 새롭게 가능한 조합을 추가해야 한다.

결국 저자의 결론은 분명하다. 모델이 좋아질수록 하니스의 역할이 사라지는 것이 아니라, 하니스가 개입해야 할 지점과 방식이 계속 달라진다는 것이다. 그리고 그 다음 유의미한 조합을 찾아내는 일이 바로 AI 엔지니어의 일이다.
